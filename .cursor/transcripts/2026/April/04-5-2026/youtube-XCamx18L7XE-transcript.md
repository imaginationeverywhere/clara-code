Se abre Claude Code, eliges tema, inicias sesión con la suscripción, autenticas, y listo: si responde, quedó bien. Salgo de Claude, limpio la terminal y sigo. Al abrir de nuevo pide ejecutar `pnpm paperclip ai` o usar enlace de invitación; en los logs del contenedor (`docker logs` + nombre) al inicio está la URL de invite. La abrimos, creamos cuenta, aceptamos el invite Bootstrap.

Aquí empieza lo divertido: configuramos la empresa. Será una web donde la gente capture y comparta prompts del día a día. La llamamos Prompt Vault: empresa que mantiene esa plataforma; usuarios se registran, comparten prompts con LLMs y agentes, y pueden rankear y calificar prompts. Algo simple.

Seguimos. Creamos el primer agente, normalmente el CEO, alineado con la visión; adaptador Claude Code; al expandir hay más plataformas. Modelo Opus 4.6 para el CEO y comprobación del entorno del adaptador — con API key prueba con esa; con Claude Code usa esa autenticación. La primera tarea por defecto es contratar al primer ingeniero y plan de hiring; yo la cambio a definir estructura del equipo y contratar a todos, quito dos ítems para conservar la parte de desglosar el roadmap en tareas concretas.

Pido explícitamente marketing con copywriter y SEO, ingeniería con líder, UI, UX, back, front, QA, etc.; un solo agente no debe cargar con todo — equipo preparado para triunfar. A ver si hace falta todo eso, pero será entretenido. Creamos y abrimos el issue. Bienvenido a Paperclip: en el dashboard ves agentes activos; solo el CEO trabajando en la tarea; un agente habilitado, cero tareas; gasto (con API keys verías coste anual y puedes poner límites).

Si abres un agente → presupuesto, puedes fijar tope. La comunicación es por tickets: nuevo issue, título, descripción, asignación — lo habitual es al CEO para que delegue. Varios proyectos; inbox donde los agentes se asignan cosas o te piden algo, por ejemplo aprobar contrataciones: el CEO propone roles, revisas capacidades, pides cambios o apruebas. Apruebo todas.

Aun así aparece un círculo rojo: tarea bloqueada — el CEO espera aprobación de todas las contrataciones. El bloqueo dice que el consejo debe aprobar las nueve contrataciones antes de empezar. Ya las aprobé; en unos minutos el CEO continúa, o puedes comentar en el ticket “aprobé las contrataciones” para reactivarlo. El CEO verifica, cierra la tarea, agentes activos, marca hecho y sigue desglosando el roadmap y delegando. Mientras tanto, en la vista de agentes entras por ejemplo al desarrollador front-end: estadísticas, instrucciones (system prompt), skills, nombre, a quién reporta, modelo (Haiku para copywriter ligero, Opus 4.6 para código), presupuesto.

La vista “org” muestra todo el equipo: CEO, CTO, desarrollo bajo CTO, VP de producto con investigador, CMO con SEO y copywriter. Iconos amarillos = inactivo, azul = ocupado; casi todo desarrollo a tope, notificaciones disparadas, texto en vivo de qué hace cada uno; el dashboard resume quién corre y qué hace. Antes de seguir: skills personalizados. En el menú Skills hay defaults de Paperclip; también “añadir skill” desde skills.sh o GitHub — copio el comando del skill de diseño front-end, pego URL o comando, añado, y el desarrollador front-end puede activarlo; igual para el copywriter.

Los agentes trabajan en Docker aislado; para tener el código fuera, crea repo en GitHub y que el equipo haga push al terminar.

Es simple: cuenta en github.com, nuevo repo “prompt vault demo”, público o privado, crear, copiar URL. Nuevo issue “Set up GitHub” al CEO: usar ese repo, buenas prácticas Git, ramas, commits, PRs al cerrar features. Antes hay que autenticar: sin credenciales GitHub dirá que no. Settings → Deploy keys → añadir; nombre “paperclip”; la clave SSH la puede generar el CEO — issue pidiendo “necesitamos deploy key, pásame la SSH”. Espero la tarjeta de soporte, el CEO genera la clave, debo añadirla al consejo. Inbox → setup GitHub → copio la clave, pego, activo “allow write”, añado (no muestro esa parte). Aviso al CEO: deploy key lista, que el equipo haga el primer push. Mientras, otros agentes avanzan: el copywriter terminó. Refresco GitHub y está el código; falta README; conectar a Vercel/Netlify es directo. Para probar, asigno al project manager: falta README, que marketing e ingeniería redacten uno limpio (no hace falta marketing para un README, pero quiero ver cómo se comporta el equipo). El CEO arranca nueva tarea; la anterior sigue: QA y front-end ocupados; nadie hace de niñera. El CEO activa a marketing y tecnología para el README conjunto.

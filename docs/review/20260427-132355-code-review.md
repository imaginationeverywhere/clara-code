## Medium-priority issues

### M1 — IDE error messages reference `clara doctor` instead of `Clara: Doctor`

**File:** `packages/ide-extension/src/http-errors.ts` (5 strings)
**Severity:** Medium (brand consistency)

Strategy briefing prompt 16 explicitly said:

> Append "Run `clara doctor`" / "Run `Clara: Doctor`" to every applicable message — make sure the wording matches the surface the user is on.

The IDE's `mapHttpErrorToString` always says `Run \`clara doctor\` for status.` — a CLI command that doesn't exist inside VS Code. A Vibe Pro using the extension sees an error notification that points them at a terminal command they can't run from the editor.

**Fix:** Replace `\`clara doctor\`` with `\`Clara: Doctor\`` in all 5 occurrences in `packages/ide-extension/src/http-errors.ts` only. CLI version is correct as-is.

**Effort:** ~2 minutes, search-and-replace in one file. Bundled into corrective prompt #20.

## Low / informational

### L1 — Code duplication between CLI and IDE for errors / tier / minutes

**Files:** `packages/cli/src/lib/{http-errors,tier-lock,minutes,minutes-exhausted}.ts` and the parallel `packages/ide-extension/src/{http-errors,tier-lock,minutes,minutes-exhausted}.ts`

The PR summary acknowledged this: "Parallel modules to CLI for extension packaging." Strings and logic are currently identical — verified line-by-line on `mapHttpError` / `mapHttpErrorToString`. Both versions reference identical user-facing copy and identical reason-keys (`tier_lock`, `minutes_exhausted`, `subscription_inactive`).

**Risk:** drift over time. If platform changes the response shape and only one side updates, CLI and IDE will give different error messages for the same gateway response — that's the brand consistency violation prompts 12 and 18 were designed to prevent.

**Recommendation:** future-only — extract to a shared workspace package (`@clara/error-mapping`) that both CLI and IDE import. NOT blocking this PR. File as tech debt.

### L2 — `deploy.ts` uses fuzzy substring check for tier-lock

**File:** `packages/cli/src/commands/deploy.ts:84`
**Code:** `if (j.reason === "tier_lock" || String(text).includes("tier"))`

The `String(text).includes("tier")` fallback could false-positive on any 403 body that happens to contain the word "tier" (e.g. "tier list," "interior," etc.) and route it to `formatTierLockMessage` with an empty payload. Low-impact because the formatter degrades gracefully when fields are missing, but it's a reliability trap.

**Recommendation:** drop the substring fallback. Trust `j.reason === "tier_lock"` exclusively. NOT blocking.

### L3 — Pre-archived prompts before review approval

**Files:** `prompts/2026/April/27/3-completed/06-*.md` … `19-*.md`

The PR pre-emptively moved 14 prompts to `3-completed/` with archival YAML frontmatter as part of the same commit that implemented them. Standard pickup-prompt convention is `1-not-started → 2-in-progress → 3-completed` only after review passes.

**Risk:** if this PR gets rejected or rolled back, the archived prompts need to be moved back. Since the prompts ARE done (code works, tests pass locally per pre-commit hook), this is more a process violation than a code violation.

**Process correction (not corrective prompt):** Future PRs from QCS1 should leave prompts in `2-in-progress/` until `/review-code` approves the PR. Move to `3-completed/` belongs in a follow-up commit (or a Phase-6 automated step in `/review-code` itself).

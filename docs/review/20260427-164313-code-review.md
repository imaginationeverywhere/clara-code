## High-priority issues

### HIGH-1 — TTS POST hits `/api/voice/tts` without Bearer auth

**File:** `packages/cli/src/lib/canonical-greeting.ts:143`
**Severity:** High (tier-enforcement bypass risk + breaks greeting on real installs)

```typescript
ttsRes = await d.fetch(`${backend.url}/api/voice/tts`, {
  method: "POST",
  headers: { "Content-Type": "application/json" },  // ← no Authorization
  body: JSON.stringify({ text: converse.reply_text }),
  ...
});
```

The backend route header explicitly says: `// POST /api/voice/tts — Clerk session or Clara API key. Text-to-speech.` (`backend/src/routes/voice.ts:44+`). This client call sends NO `Authorization` header.

**Two failure modes:**
- If backend correctly enforces auth: every greeting attempt 401s. Voice never plays for real users.
- If backend silently lets unauthenticated calls through: tier enforcement bypassed. A scripted attacker could rip unlimited TTS minutes via `curl -X POST .../api/voice/tts -d '{"text":"..."}'`. Strategy briefing locked rule #13: "Every command checks tier server-side via the gateway. Never client-side gating only — clients can be patched."

**Compare to** `packages/cli/src/lib/agents-api.ts:133` (correct):
```typescript
headers: {
  Authorization: await authHeader(),
  "Content-Type": "application/json",
},
```

**Fix:** Add `Authorization: \`Bearer \${token}\`` to the TTS call. Source the token from `readClaraCredentials()` + `pickBearerToken()` like the cognitive verbs and `agents-api.configureAgent` already do.

### HIGH-2 — Backend production code still uses `HERMES_*` env var names

**File:** `backend/src/routes/voice.ts` (8+ references)
**Severity:** High (locked-rule violation in shipped code)

Strategy briefing locked rule (every QCS1 prompt referenced this): *"Env vars: `CLARA_GATEWAY_URL`, `CLARA_BRAIN_URL`, `CLARA_BACKEND_URL`. **Never `HERMES_*`.**"*

The CLI/IDE rename landed via PR #64 (`be4402b2`). The backend rename **never happened**. Production code still reads:

```typescript
backend/src/routes/voice.ts:44:  const hermes = process.env.HERMES_GATEWAY_URL?.trim();
backend/src/routes/voice.ts:51:  const k = process.env.HERMES_API_KEY?.trim();
backend/src/routes/voice.ts:57:  const HERMES_TIMEOUT_MS = 150_000;
backend/src/routes/voice.ts:262:  // Real mode: proxies to HERMES_GATEWAY_URL/voice/stt (Whisper on Modal, cp-team owned).
backend/src/routes/voice.ts:304:  logger.error("HERMES_API_KEY is not set — refusing to proxy to Modal");
backend/src/routes/voice.ts:311:  { timeout: HERMES_TIMEOUT_MS, headers: hermesHeaders() },
backend/src/routes/voice.ts:341:  // Real mode: proxies to HERMES_GATEWAY_URL/voice/tts ...
backend/src/routes/voice.ts:381:  logger.error("HERMES_API_KEY is not set — refusing to proxy to Modal");
```

Plus 9+ matches in `backend/src/__tests__/routes/voice.test.ts`.

**Why it matters:** The internal-names rule isn't just for end-user surfaces. Logs surface in CloudWatch where developer eyes (some non-founder) read them. Error messages with `"HERMES_API_KEY is not set"` leak the internal name to anyone with log access. SSM parameters reference these names. Every new dev onboarded sees them.

**Fix:**
- Rename env var reads: `process.env.HERMES_GATEWAY_URL` → `process.env.CLARA_GATEWAY_URL`
- Rename: `HERMES_API_KEY` → `CLARA_GATEWAY_API_KEY` (or similar Clara-prefixed name)
- Update SSM parameters at `/quik-nation/clara-code/HERMES_GATEWAY_URL` etc. to the new names (or set new SSM keys + update App Runner env mapping).
- Update test files in `__tests__/routes/voice.test.ts` to match.
- Update log strings: `"HERMES_API_KEY is not set"` → `"Gateway API key is not set"`.

This is a bigger change than CRIT-1 — touches infra (SSM, App Runner config). Could be a follow-up sprint task rather than urgent corrective. Surfacing as HIGH because the locked rule was unambiguous and Sprint-4 prompts will continue to assume the rename is done.

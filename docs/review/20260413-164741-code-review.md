## Medium Priority Issues

### M1 — `git add "prompts/"` is too broad

**Files:** `.claude/commands/queue-prompt.md` (last block), `.claude/commands/review-code.md` (Step 3b)

Both new blocks use `git add "prompts/"` which stages the **entire prompts directory tree**, not just the newly written file. If the working tree has unrelated staged or unstaged changes in `prompts/` from a previous partial run (e.g., a `2-in-progress/` file that got stranded), those changes get bundled into the commit unintentionally.

**Fix (both files):** Scope the add to the specific date subdirectory and the specific file:

```bash
# In queue-prompt.md:
git add "prompts/${YEAR}/${MONTH}/${DAY}/1-not-started/"

# In review-code.md (Step 3b):
git add "prompts/${YEAR}/${MONTH}/${DAY}/1-not-started/${NEXT_NUM}-fix-${SLUG}.md"
```

### M2 — No error handling on `git push` failure

**Files:** `.claude/commands/queue-prompt.md`, `.claude/commands/review-code.md`

Both push steps lack error detection. If `git push origin "$BRANCH"` fails (expired credentials, network issue, diverged remote, protected branch), the current spec silently echoes `"✓ Prompts pushed to GitHub: $BRANCH"` regardless. A queued prompt that was never actually pushed will leave QCS1 waiting for work that doesn't exist on the remote.

**Fix:** Add exit-code checking and an explicit failure message:

```bash
if git push origin "$BRANCH"; then
  echo "✓ Prompts pushed to GitHub: $BRANCH"
else
  echo "⚠️  Push failed — prompts are local only. Run: git push origin $BRANCH manually."
fi
```

## CRITICAL Issues (must fix before any production traffic)

### 1. Auth-bypass via silent fallback to "free" tier тАФ FIXED ABOVE
**Original problem:** Across ~30 callsites the code did `req.claraUser?.tier ?? "free"`. Any
unauthenticated request that bypassed an auth guard silently inherited the most permissive
"free" defaults. Combined with the no-free-tier rule, this both leaked permissions AND violated
canonical pricing.

**Resolution:** All defaults moved to `"basic"`. `PlanTier` no longer admits `"free"`. The
TS compiler now blocks new `"free"` defaults at compile time. `user-api-key.ts` now gates
on `sub.status !== "active"` instead of tier sniffing.

**Residual risk:** `?? "basic"` still gives unauth callers paid-tier feature access if a route
ever forgets `requireAuth`. **Recommend** introducing a `tier: PlanTier | null` shape and forcing
explicit `tier == null тЖТ 401` at every entry point. Tracked in corrective prompt.

### 2. Marketplace API leak тАФ `GET /api/talents/agent/:agentId` skips ownership check
**File:** `backend/src/routes/talents.ts:109-117`
**Problem:** The route calls `talentService.listAgentTalents(req.params.agentId)` with no check
that the agent belongs to `req.claraUser.userId`. Any authenticated user can probe what Talents
ANY other user's agent has by guessing/scraping agent IDs.
**Compare:** `attach()` (line 80) properly checks `UserAgent.findOne({ where: { id, userId } })`.
The list endpoint must too.
**Fix:** Add ownership check before calling `listAgentTalents`, or replace with
`getTalentBlockForUserAgent` which already validates ownership.
**Severity:** CRITICAL (cross-tenant info disclosure on a marketplace).

### 3. Open-redirect on Stripe checkout URLs
**File:** `backend/src/routes/billing.ts:43-44, 76-77`
**Problem:** `success_url` and `cancel_url` come from `req.body` and are passed verbatim to
Stripe. A malicious POST `/api/billing/checkout` with `success_url: "https://evil.example/"`
will redirect the user to attacker-controlled territory after a successful payment, leaking
the Stripe `session_id` query parameter.
**Fix:** Validate that supplied URLs share the configured `frontendUrl` host, or remove
the body parameter entirely and only use the configured default.
**Severity:** CRITICAL (security).

### 4. `talentService.acquire()` is non-atomic
**File:** `backend/src/services/talent.service.ts:93-168`
**Problem:** The flow does (a) wallet debit, (b) publisher credit, (c) purchase row insert,
(d) library row insert/update тАФ each in independent transactions. A failure between (a) and
(d) leaves the user charged but unentitled, or entitled and undercharged depending on order.
**Fix:** Wrap the entire flow in a single `sequelize.transaction(async t => { ... })` and
plumb `t` into wallet calls. Add a unique constraint on
`agent_talent_purchases (user_id, talent_id, idempotency_key)` for at-least-once semantics.
**Severity:** CRITICAL (revenue integrity).

### 5. `talentService.attach()` has TOCTOU race on tier cap
**File:** `backend/src/services/talent.service.ts:180-211`
**Problem:** Reads attachment count, compares to cap, then inserts. Two concurrent attach
requests from the same user can both pass the count check and create two rows, exceeding the
cap. Per memory rule
[`feedback_talents_abilities_naming_strict.md`], cap enforcement is contractual.
**Fix:** Use `SELECT ... FOR UPDATE` inside a transaction, or rely on a partial unique index
that limits attachments per `user_agent_id` and let DB enforce.
**Severity:** CRITICAL (subscription tier evasion).

## HIGH Issues

### 6. Wallet has no transaction ledger
**File:** `backend/src/services/wallet.service.ts`
**Problem:** Debits/credits log to `logger.info` but there is no `wallet_transactions` table.
You cannot reconcile against Stripe, can't repay disputes, can't audit fraud.
**Fix:** Add `wallet_transactions (id, user_id, amount, type, reference, idempotency_key,
created_at)` migration; persist a row inside the wallet's existing transaction.

### 7. Wallet debit has no idempotency
**File:** `backend/src/services/wallet.service.ts:39-63`
**Problem:** A retried request (network blip, double-click) debits the user twice. The
`_reference` parameter exists but is unused (note the `_` prefix). No unique constraint, so
duplicate calls pass.
**Fix:** Make `reference` required, persist into `wallet_transactions` with a unique constraint,
treat duplicate insert as a successful no-op.

### 8. `agent_talent_purchases` permits duplicate Stripe webhooks
**File:** `backend/migrations/040_harness_talent_catalog_and_wallet.sql:66-77`
**Problem:** No `UNIQUE (stripe_payment_id)` or idempotency key. A retried webhook creates a
second purchase row.
**Fix:** Add migration `043_unique_stripe_payment_on_talent_purchases.sql` with a partial
unique index on non-null `stripe_payment_id`.

### 9. Operation-credit race between `canUseOperationCredits` and `applyOperationCreditUsage`
**File:** `backend/src/services/operation-credit.service.ts:42-126`
**Problem:** Caller does check-then-apply with a network round-trip in between. Concurrent
requests can both pass the check and then both apply, exceeding monthly cap.
**Fix:** Either combine into a single `consumeCredit()` that locks the row and decrements
atomically, or use `UPDATE ... WHERE credits_used + ? <= cap RETURNING ...` and check rows
affected.

### 10. Voice usage events are fire-and-forget
**File:** `backend/src/services/abuse-protection.service.ts:154-166`
**Problem:** `UsageEvent.create({...}).catch(...)` does not await. If persistence fails, the
audit row is lost. This data is the COGS source of truth тАФ losing events means under-reporting
infrastructure cost.
**Fix:** `await` the create, and on failure write to a dead-letter queue (Redis list or a
dedicated table) for reconciliation.

### 11. Knowledge content from third-party Talents is concatenated raw into the agent prompt
**File:** `backend/src/services/talent.service.ts:230-240`
**Problem:** `getTalentBlockForUserAgent` joins `t.knowledgeContent` directly into the system
prompt. Once third-party publishers can submit Talents (per
prompt 14 spec), a malicious publisher can inject `\n\n# IGNORE PREVIOUS INSTRUCTIONS\n...`
into a Talent's knowledge and pivot the agent.
**Fix:** Wrap each block in a sandboxed delimiter the agent's system prompt instructs it to
treat as data only (e.g., XML tags `<TALENT_KNOWLEDGE id="..." trust="external">...</...>`),
combined with a content scan for prompt-injection patterns at publish time.

### 12. Wallet balance has no overdraft protection at the DB layer
**File:** `backend/migrations/040_...sql:5-10`
**Problem:** `balance_usd NUMERIC(12,2) DEFAULT 0` permits negative values. Service code checks
`bal >= amount` but a buggy callsite or a Sequelize `decrement` could go negative.
**Fix:** Add `CHECK (balance_usd >= 0)` constraint.

## MEDIUM Issues

### 13. `user_usage` schema only retains current month
**File:** `backend/migrations/038_user_usage_and_usage_events.sql:4-15`
**Problem:** PRIMARY KEY is `user_id` alone. Month rollover overwrites prior month's row.
Historical reporting depends entirely on `usage_events`. If events are dropped (see HIGH #10)
data is gone.
**Fix:** Either change PK to `(user_id, month_key)` to retain rows, or add a job that snapshots
prior month into a history table on the 1st.

### 14. `is_public BOOLEAN DEFAULT TRUE` on `agent_talent_catalog`
**File:** `backend/migrations/040_...sql:26`
**Problem:** Marketplace Talents default to public on insert. User-published Talents should
default private until reviewed; defaulting public risks accidental marketplace listings.
**Fix:** `DEFAULT FALSE`. Curated/seeded Talents explicitly opt in via `INSERT ... is_public=true`.

### 15. `classifyOperation` is keyword-fragile pricing logic
**File:** `backend/src/services/operation-weights.ts:35-59`
**Problem:** Weight (= billing) depends on regex matches against the user's voice transcript.
"Make a soccer agent" тЖТ `light` (1 credit). "Create a soccer agent" тЖТ `agent_build` (20 credits).
Same intent, different price. Customer-visible.
**Fix:** Either make the LLM emit an `intent_class` token in its response that the router
trusts, or move classification post-execution based on what was actually done.

### 16. No FK constraints on `user_id` columns
**Files:** migrations 038, 039, 040
**Problem:** `user_id` is `VARCHAR(255)` (Clerk ID, intentional), but no `REFERENCES users(id)`
which means orphaned rows on user delete. Acceptable for the Clerk pattern but worth noting
for future cleanup logic.

### 17. `EJECTION_CAPS` and `PLAN_LIMITS.runtimeAgentBuildsPerMonth` are duplicate sources of truth
**Files:** `services/ejection.service.ts:16-22`, `services/plan-limits.ts:18`
**Problem:** `EJECTION_CAPS` uses 1/3/6/12 and `PLAN_LIMITS.runtimeAgentBuildsPerMonth` now also
uses 1/3/6/12. They're the same numbers in two places тАФ drift inevitable.
**Fix:** Remove `EJECTION_CAPS`, source from `PLAN_LIMITS[tier].ejectionsPerMonth` (rename or
add).

### 18. CSRF unprotected on billing routes
**File:** `backend/src/routes/billing.ts`
**Problem:** Mutating POSTs (`/checkout`, `/cancel`, `/upgrade`, `/downgrade`, `/refund`) accept
JSON without CSRF tokens. Clerk session is the only auth. Cross-site requests with a Clerk
cookie can trigger upgrades/cancels.
**Fix:** Add a CSRF middleware (double-submit cookie or origin check) for state-changing
billing endpoints.

### 19. Hardcoded production URLs
**Files:** `services/operation-credit.service.ts:6`, others
**Problem:** `const PRICING_URL = "https://claracode.ai/pricing"` baked into source. Should be
`process.env.FRONTEND_URL + "/pricing"` (the pattern already exists in
`config/models.ts:pricingUrl()`).
**Fix:** Reuse the helper, kill the constant.

### 20. Voice-limit middleware is a deprecated stub but still imported by tests
**File:** `backend/src/middleware/voice-limit.ts`, plus 2 test files
**Problem:** Marked `@deprecated`, kept as a no-op as a guard rail against re-introducing a
voice cap. Source code no longer imports it, but tests do, so the deprecation diagnostic fires
on every CI run.
**Fix:** Either delete the file (it's no longer protecting anything) or move the guard rail
into a CODEOWNERS / pre-commit script that flags resurrection of voice caps.

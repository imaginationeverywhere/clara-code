## Critical Issues

### CRIT-01 — IP Protection Violation: Hardcoded Modal URL in Desktop Client
**File:** `desktop/src/voice-overlay.ts:3-4`
**Severity:** CRITICAL — Violates IP protection rule

```ts
// ❌ CURRENT — exposes Modal endpoint in client-side bundle
const TTS_URL =
  "https://info-24346--clara-voice-server-voiceserver-fastapi-app.modal.run/voice/tts";
```

The desktop voice overlay calls Modal TTS directly from the browser-loaded JavaScript. The Modal deployment URL is exposed in the client bundle, visible to any user who inspects DevTools. This violates the established IP protection rule: **"No server URLs/scripts in client packages; all API calls through Next.js `/app/api/` routes."**

**Fix:**
1. Remove `TTS_URL` constant from `voice-overlay.ts`
2. Change the `fetch` call to POST to `/api/voice/tts` (the existing server-side proxy)
3. The Tauri `capabilities/default.json` already allows `https://claracode.ai` — local dev needs `http://localhost:PORT/api/voice/tts`

```ts
// ✅ CORRECT
const response = await fetch("/api/voice/tts", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ text: DEFAULT_TTS_TEXT }),
});
```

This URL is the dev ngrok tunnel. It will fail for:

- Any device not hitting the dev environment

export const CLARA_API_BASE =

Then set `EXPO_PUBLIC_CLARA_API_BASE=https://api.claracode.com` in production EAS build profile.

## Medium Priority Issues

### MED-01 — Voice Greet Endpoint Has No Rate Limiting
**File:** `frontend/src/app/api/voice/greet/route.ts`

The `/api/voice/greet` endpoint makes a server-side call to the Modal voice server. There is no rate limit or caching — any bot or crawler hitting the marketing page will trigger a Modal TTS call for every request. The TTS route has a `500 char` text limit, but no per-IP or per-session rate limit.

**Fix:** Add a `Cache-Control: public, max-age=300` header (already done for the response stream) and add a server-side request cache keyed on the message hash to deduplicate identical requests within the TTL window.

### MED-03 — CLI Config Hardcodes Modal URLs in User-Facing Defaults
**Files:** `packages/cli/src/lib/config.ts:16`, `packages/cli/src/commands/greet.ts:8`, `packages/cli/src/commands/tui.tsx:10`

The CLI ships with Modal deployment URLs as hardcoded defaults. While CLI tools may include defaults, these are internal Modal function URLs that will break if the deployment is updated or scaled. The URL format (`info-24346--`) is a deployment-specific identifier that changes on re-deploy.

**Fix:** Read these from `process.env.CLARA_GATEWAY_URL` / `CLARA_VOICE_URL` with sensible fallback to the current URL. Document the env vars in the CLI's `--help` output. A future improvement would be auto-discovery from the Clara platform API.

### MED-04 — Mobile Has No Timeout on API Calls
**File:** `mobile/lib/clara-api.ts`

All `fetch` calls in `clara-api.ts` have no timeout. On a slow connection, the voice screen will hang indefinitely showing the processing spinner. On iOS, background fetch is also not handled for the audio download case.

**Fix:** Wrap fetch calls with `AbortController` and a 30-second timeout:
```ts
const controller = new AbortController();
const timeout = setTimeout(() => controller.abort(), 30_000);
try {
  const res = await fetch(url, { signal: controller.signal, ...opts });
  clearTimeout(timeout);
  return res;
} catch (e) {
  if (e instanceof Error && e.name === 'AbortError') {
    throw new Error('Request timed out');
  }
  throw e;
}
```

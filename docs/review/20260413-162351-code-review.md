## High Priority Issues

### H1 — `VoiceSessionImpl` silently swallows initialization errors

**File:** `packages/sdk/src/client.ts` — `VoiceSessionImpl` constructor
**Impact:** Silent wrong behavior in production — callers get confusing 404s with empty session ID

The constructor calls `void this.create()`. When `create()` hits an HTTP error, it calls `this.settleReady()` (resolving `ready` to fulfilled) then throws. Because of `void`, that throw becomes an unhandled promise rejection. Any caller who then calls `send()` will `await this.ready` (which succeeded), pass the `this.closed` guard (still `false`), and fire a request with `this._id === ""` — producing a URL like `/v1/voice/sessions//messages` and a non-obvious 404.

**Fix:**
```typescript
// Add reject path to the ready promise
private readonly rejectReady!: (err: Error) => void;

constructor(private readonly config: ClaraConfig) {
  let settle!: () => void;
  let reject!: (err: Error) => void;
  this.ready = new Promise<void>((res, rej) => {
    settle = res;
    reject = rej;
  });
  this.settleReady = settle;
  this.rejectReady = reject;
  void this.create();
}

private async create(): Promise<void> {
  try {
    // ... existing fetch logic (unchanged) ...
    this._id = data.id;
    this.settleReady();
  } catch (err) {
    this.rejectReady(err instanceof Error ? err : new Error(String(err)));
  }
}
```

### H2 — Health endpoint async handler unprotected in Express 4

**File:** `backend/src/server.ts:44`
**Impact:** Hung health probes → ECS marks task unhealthy → task replacement loop

Express 4 does not automatically catch rejected promises from async route handlers. If `testConnection({ silent: true })` throws (network timeout, pool error), the request hangs indefinitely. This endpoint is hit on every ECS health probe interval.

**Current (vulnerable):**
```typescript
app.get("/health", async (_req, res) => {
  const dbOk = await testConnection({ silent: true });
  res.json({ status: "ok", db: dbOk ? "connected" : "error", service: "clara-code-backend" });
});
```

**Fixed:**
```typescript
app.get("/health", async (_req, res) => {
  try {
    const dbOk = await testConnection({ silent: true });
    res.json({ status: "ok", db: dbOk ? "connected" : "error", service: "clara-code-backend" });
  } catch {
    res.status(503).json({ status: "error", db: "unreachable", service: "clara-code-backend" });
  }
});
```

## Medium Priority Issues

### M1 — WebSocket toggle race condition in IDE extension

**File:** `ide/clara-code/extensions/clara-voice/src/extension.ts` — `toggle()` (~line 50)

`isRecording` is set `true` only inside the `"open"` event callback. Between `new WebSocket(wsUrl)` and the `open` event firing, `isRecording` is still `false`. A second toggle during this window creates a second socket without closing the first. After both open, `isRecording` is set `true` twice, only one socket is tracked, and the first socket leaks.

**Fix:** Add `isConnecting` boolean set synchronously before the constructor, cleared on `open` or `error`.

### M2 — Missing `Accept: text/event-stream` header on agent stream

**File:** `packages/sdk/src/client.ts` — `streamAgentChunks` (~line 164)

`streamChunks` correctly sends `Accept: "text/event-stream"`. `streamAgentChunks` does not. If Hermes uses content negotiation, agent streams will silently receive JSON instead of SSE and produce no output.

**Fix:** Add `Accept: "text/event-stream"` to `streamAgentChunks` headers (mirror `streamChunks` pattern).

### CI Fix Sequence: `e6db1a9f` → `28711cb6` → `d44146d2` → `2f7afe9d` → `dd4d9c8e`

Root problem: `npm workspaces` hoists `next` to the repo-root `node_modules/`. In Cloudflare Pages CI, Turbopack defaults its lookup boundary to `frontend/` — so it can't find `next/package.json` without help. Five iterative commits converged on the correct fix.

**Attempts (e6db1a9f → 2f7afe9d):**
1. `turbopack.root = process.cwd()` — wrong dir (already inside frontend/)
2. `turbopack.root = repoRoot` + `outputFileTracingRoot = repoRoot` — conflicted with Vercel auto-inferred root
3. Moved `outputFileTracingRoot` to top-level — still conflicted
4. Removed `outputFileTracingRoot`, kept `turbopack.root` — still caused path doubling

**Final fix (`dd4d9c8e`):**
```yaml
# .github/workflows/deploy-pages.yml
- name: Link next into frontend/node_modules (resolve hoisting)
  run: |
    mkdir -p frontend/node_modules
    ln -sfn "$(pwd)/node_modules/next" frontend/node_modules/next
```
```ts
// frontend/next.config.ts — reverted to clean state, no turbopack.root needed
```

The symlink approach is correct: it satisfies Turbopack's default boundary lookup without conflicting with `outputFileTracingRoot` auto-inference. The `next.config.ts` returns to its original clean form.

# Mobile Agent Notes Capture → Update Queue

**TARGET REPO:** imaginationeverywhere/clara-code (platform) + used by mobile apps embedding Clara
**Priority:** P1 — needed for full Heru parity on mobile
**Packages:** `backend/`, `packages/sdk/`
**Depends on:** prompt 23 (SITE_OWNER interaction patterns)
**Milestone:** When SITE_OWNERs talk to agents from a MOBILE app (iOS/Android), they can't trigger code changes or site modifications on the fly — the app is already compiled and in App Store / Play Store. Instead, the agent captures the request as a structured note, queues it for the next app submission cycle, and drafts the change spec for Clara's harness team to execute. SITE_OWNER gets acknowledgement; nothing breaks; updates ship in the next release.

---

## The Flow

```
SITE_OWNER (on mobile app): "Add a tipping screen after checkout."
  ↓
Runtime agent on mobile captures → sends to Clara backend
  ↓
Agent acknowledges: "Got it. I'll draft a spec and queue it for your next release. You can review it in your dashboard."
  ↓
Backend: creates update_request row with:
  - voice transcript
  - structured spec (generated by agent after transcript)
  - priority guess
  - affected screens
  ↓
Dashboard (web or tablet): SITE_OWNER reviews + approves + slots into next release
  ↓
When approved + scheduled: harness team on Clara Code picks up spec + ships change
  ↓
Mobile app submission → App Store review → released → SITE_OWNER notified
```

---

## Why This Exists

Mobile apps can't hot-update like web. But SITE_OWNERs still need a way to say "fix this" or "add that" naturally via voice while using their app. This prompt builds the note-capture-and-queue system that bridges mobile voice → Clara's build team.

## Distinction from Web (prompt 23)

| | Web (prompt 23) | Mobile (this prompt) |
|--|-----------------|----------------------|
| Change takes effect | Immediately via SITE_OWNER instruction overlay | Next app release |
| Scope | Content, rules, agent behavior | Needs new code / UI / feature |
| Response to SITE_OWNER | "Done — your change is live." | "Queued for next release. Review spec in dashboard." |
| Platform standards | Enforced at overlay apply time | Enforced at spec review / approval time |

## Part 1 — Migration

**File:** `backend/migrations/024_mobile_update_queue.sql`

```sql
CREATE TABLE IF NOT EXISTS mobile_update_requests (
  id                  UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
  deployment_id       UUID         NOT NULL REFERENCES site_agent_deployments(id) ON DELETE CASCADE,
  site_owner_user_id  VARCHAR(255) NOT NULL,
  heru_slug           VARCHAR(100) NOT NULL,
  platform            VARCHAR(20)  NOT NULL CHECK (platform IN ('ios','android','both')),

  raw_voice_transcript TEXT        NOT NULL,
  agent_interpretation TEXT        NOT NULL,               -- agent's restatement
  structured_spec      JSONB       NOT NULL,               -- {title, description, acceptance_criteria, affected_screens, priority_guess}
  priority             VARCHAR(20) NOT NULL DEFAULT 'normal',  -- low / normal / high / urgent

  status               VARCHAR(30) NOT NULL DEFAULT 'pending_review',  -- pending_review / approved / in_progress / shipped / rejected
  target_release       VARCHAR(50),                           -- e.g., "v1.2.0" or "2026-05"
  approved_at          TIMESTAMPTZ,
  shipped_at           TIMESTAMPTZ,
  rejected_reason      TEXT,
  created_at           TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX IF NOT EXISTS idx_mobile_reqs_deployment
  ON mobile_update_requests (deployment_id, status);
CREATE INDEX IF NOT EXISTS idx_mobile_reqs_owner
  ON mobile_update_requests (site_owner_user_id, status, created_at DESC);
```

## Part 2 — Note Capture Service

**File:** `backend/src/services/mobile-note-capture.service.ts`

```typescript
import { MobileUpdateRequest, SiteAgentDeployment } from "@/models";
import { hermesClient } from "./hermes-client.service";
import { platformStandards } from "./platform-standards.service";
import logger from "@/lib/logger";

export interface NoteCaptureInput {
  deploymentId: string;
  siteOwnerUserId: string;
  platform: "ios" | "android" | "both";
  rawVoiceTranscript: string;
}

export class MobileNoteCaptureService {
  async captureFromVoice(input: NoteCaptureInput): Promise<MobileUpdateRequest> {
    const deployment = await SiteAgentDeployment.findByPk(input.deploymentId);
    if (!deployment) throw new Error("deployment_not_found");

    // Use the agent itself to interpret the voice note into a structured spec
    const spec = await this.interpretAsSpec(input.rawVoiceTranscript, deployment);

    // Validate against platform standards (same as web SITE_OWNER flow)
    const validation = await platformStandards.validate(
      spec.description,
      spec.category ?? "behavior",
    );
    if (!validation.approved) {
      throw new Error(`platform_rejected:${validation.rejectionReason}`);
    }

    return await MobileUpdateRequest.create({
      deploymentId: input.deploymentId,
      siteOwnerUserId: input.siteOwnerUserId,
      heruSlug: deployment.heruSlug,
      platform: input.platform,
      rawVoiceTranscript: input.rawVoiceTranscript,
      agentInterpretation: spec.restatement,
      structuredSpec: spec,
      priority: spec.priority_guess,
      status: "pending_review",
    });
  }

  private async interpretAsSpec(transcript: string, deployment: any): Promise<any> {
    const promptToAgent = `
You are interpreting a SITE_OWNER's voice note for mobile app update queue.
Transcript: "${transcript}"

Output JSON with fields:
{
  "restatement": "one-sentence clear restatement of what they want",
  "title": "short PR-title-style (under 60 chars)",
  "description": "full description for engineers",
  "acceptance_criteria": ["bullet list of criteria"],
  "affected_screens": ["screen names"],
  "priority_guess": "low|normal|high|urgent",
  "category": "content|behavior|new_feature|bug_fix"
}
`;

    const response = await hermesClient.inference(
      { model: "kimi_k2", prompt: promptToAgent, maxTokens: 1024 },
      { userId: deployment.siteOwnerUserId, tier: "basic", taskType: "reasoning", inputTokenEstimate: 1000, userHasDeepestPlugin: false, explicitPremiumRequest: false },
    );

    try {
      return JSON.parse(response.text);
    } catch {
      return {
        restatement: transcript,
        title: transcript.slice(0, 60),
        description: transcript,
        acceptance_criteria: [],
        affected_screens: [],
        priority_guess: "normal",
        category: "behavior",
      };
    }
  }

  async approve(requestId: string, approverId: string, targetRelease?: string): Promise<void> {
    await MobileUpdateRequest.update(
      { status: "approved", approvedAt: new Date(), targetRelease: targetRelease ?? null },
      { where: { id: requestId, siteOwnerUserId: approverId } },
    );
  }

  async reject(requestId: string, approverId: string, reason: string): Promise<void> {
    await MobileUpdateRequest.update(
      { status: "rejected", rejectedReason: reason },
      { where: { id: requestId, siteOwnerUserId: approverId } },
    );
  }

  async markShipped(requestId: string): Promise<void> {
    await MobileUpdateRequest.update(
      { status: "shipped", shippedAt: new Date() },
      { where: { id: requestId } },
    );
  }

  async listPendingForOwner(siteOwnerUserId: string) {
    return await MobileUpdateRequest.findAll({
      where: { siteOwnerUserId, status: "pending_review" },
      order: [["createdAt", "DESC"]],
    });
  }
}

export const mobileNoteCapture = new MobileNoteCaptureService();
```

## Part 3 — Routes

**File:** `backend/src/routes/mobile-updates.ts`

```typescript
import { Router } from "express";
import { requireClaraOrClerk } from "@/middleware/api-key-auth";
import { mobileNoteCapture } from "@/services/mobile-note-capture.service";

const router = Router();

router.post("/capture", requireClaraOrClerk, async (req, res) => {
  try {
    const result = await mobileNoteCapture.captureFromVoice({
      deploymentId: req.body.deployment_id,
      siteOwnerUserId: req.claraUser!.userId,
      platform: req.body.platform,
      rawVoiceTranscript: req.body.transcript,
    });
    res.status(201).json({
      request: result,
      agent_response: `Got it. "${result.agentInterpretation}" — queued for your next release. Review in your dashboard.`,
    });
  } catch (err) {
    const message = err instanceof Error ? err.message : "error";
    res.status(400).json({ error: message });
  }
});

router.get("/pending", requireClaraOrClerk, async (req, res) => {
  const pending = await mobileNoteCapture.listPendingForOwner(req.claraUser!.userId);
  res.json({ pending });
});

router.post("/:id/approve", requireClaraOrClerk, async (req, res) => {
  await mobileNoteCapture.approve(req.params.id, req.claraUser!.userId, req.body.target_release);
  res.json({ approved: true });
});

router.post("/:id/reject", requireClaraOrClerk, async (req, res) => {
  await mobileNoteCapture.reject(req.params.id, req.claraUser!.userId, req.body.reason);
  res.json({ rejected: true });
});

router.post("/:id/shipped", requireClaraOrClerk, async (req, res) => {
  // Called by Clara ops / CI when the app release ships
  await mobileNoteCapture.markShipped(req.params.id);
  res.json({ marked_shipped: true });
});

export default router;
```

## Part 4 — Mobile SDK Hook (for React Native Herus)

**File:** `packages/sdk/src/mobile/useAgentNoteCapture.ts`

```typescript
export function useAgentNoteCapture(deploymentId: string) {
  return {
    captureVoiceNote: async (transcript: string, platform: "ios" | "android") => {
      const res = await fetch(`${CLARA_BACKEND_URL}/api/mobile-updates/capture`, {
        method: "POST",
        headers: authHeaders(),
        body: JSON.stringify({ deployment_id: deploymentId, platform, transcript }),
      });
      return await res.json();
    },
  };
}
```

Every Heru mobile app imports this once and calls it when a SITE_OWNER voices a note.

## Part 5 — Agent Response Pattern

When SITE_OWNER voices a note on mobile, the agent should:
1. Acknowledge warmly: "Got it."
2. Restate what they heard: "You want a tipping screen after checkout."
3. Set expectations: "I've queued this for your next release and drafted a spec."
4. Invite review: "You can approve it in your dashboard — we'll ship in roughly [release window]."

System prompt addition for runtime agents on mobile:

```
[MOBILE CAPTURE MODE]
When the SITE_OWNER requests a change that requires code, UI, or new features:
  1. Acknowledge warmly
  2. Restate the request in one clear sentence
  3. Tell them it's queued for next release
  4. Tell them they can review the spec in their dashboard
Do NOT pretend to make changes that require app submission.
Do NOT say "done" or "I've updated it" — those are misleading on mobile.
```
